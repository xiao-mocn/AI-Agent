第 10 课 · 知识 + 技能

流式输出——让 AI 逐字"打"出来

用户不用等完整回复，AI 边想边显示

现在 vs 目标

非流式（现在）
用户发消息 → 等 5 秒 → 完整回复一下子弹出来。
感觉像"系统在卡顿"，体验差。
流式（目标）
用户发消息 → 第 1 个字立即出现 → 逐字追加 → 完成。
像真人打字一样，用户更快开始阅读。

流式是怎么工作的

非流式时，DeepSeek 把整段回复一次性发回来。 流式时，DeepSeek 每生成一个词就发一个 chunk（数据块）， 你的后端把这些 chunk 逐个转发给前端，前端逐个拼接到聊天界面上。

一个 chunk 长这样
{"choices":[{"delta":{"content":"闭"}}]}
{"choices":[{"delta":{"content":"包"}}]}
{"choices":[{"delta":{"content":"是"}}]}
{"choices":[{"delta":{"content":" "}}]}
{"choices":[{"finish_reason":"stop"}]}

每个 chunk 只有一两个字，后端不收集完整回复——收到一个就转发一个。 最后一个 chunk 带 finish_reason: "stop"，表示说完了。

和之前的区别：非流式时后端用 response.choices[0].message.content 拿到完整文案再返回。 流式时后端不收集文案，只负责"管传"——后端变成一个管道，把 DeepSeek 的 chunks 原样推给前端。

动手：三处改动

1
后端：把路由改成流式转发
改动 src/index.ts 里 /api/chat 路由的 try 块：
src/index.ts — 完整的 try 块
try {
  const stream = await client.chat.completions.create({
    model: 'deepseek-chat',
    messages: history,
        stream: true,  // <-- 唯一新增的参数
  })

    // 把 ReadableStream 转成 SSE 响应
    let fullReply = ''
    return new Response(
      new ReadableStream({
        async start(controller) {
          for await (const chunk of stream) {
            const delta = chunk.choices[0]?.delta?.content
            if (delta) {
              fullReply += delta
              controller.enqueue(new TextEncoder().encode(`data: ${JSON.stringify({ delta })}\n\n`))
            }
          }
          // 发送结束标记
          controller.enqueue(new TextEncoder().encode(`data: [DONE]\n\n`))
          controller.close()
          // 写入完整回复到数据库（用于历史还原）
          insertMsg.run(sessionId, 'assistant', fullReply)
        },
      }),
      {
        headers: { 'Content-Type': 'text/event-stream' },
      }
    )

前端：改造 fetch 以消费流
改动 ai-frontend/src/App.vue 里的 send() 函数：
App.vue — 替换 fetch 部分
const res = await fetch(`${API_URL}/api/chat`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ message: text, sessionId }),
})
if (!res.ok) throw new Error('请求失败')

// 创建一个空的 assistant 消息，边读边填充
messages.value.push({ role: 'assistant', content: '' })
const lastMsg = messages.value[messages.value.length - 1]

const reader = res.body!.getReader()
const decoder = new TextDecoder()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const text = decoder.decode(value)
  // SSE 格式：每行一个 event
  const lines = text.split('\n')
  for (const line of lines) {
    if (!line.startsWith('data: ')) continue
    const payload = line.slice(6)
    if (payload === '[DONE]') break
    const { delta } = JSON.parse(payload)
    lastMsg.content += delta  // 追加到消息气泡里
  }
}

验证流式效果

同时启动前后端，在浏览器聊天界面里发一条长问题（比如"写一个快速排序的 TypeScript 实现"）。 观察回复是否在打字阶段逐字展开，而不是一次性出现。

关键点：后端不再返回 c.json(...)，而是返回一个 new Response(ReadableStream, ...)。 因为 streaming 不匹配 Hono 的 JSON 模式，需要用 Web 标准的 Response 对象。 这也是为什么 CORS 中间件很重要——它对这个新 Response 同样生效。
