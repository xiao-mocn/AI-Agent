错误处理——让接口不崩溃

两处改动，让你的 AI 接口从"跑起来"变成"跑稳"

现在的接口有什么问题？

你现在的 /api/chat 接口已经能正常工作了——但它假设一切都会顺利。 真实世界里有两类问题会让它直接崩溃：

  • 坏输入：用户没传 sessionId，或者 message 是空字符串
  • 下游故障：DeepSeek API 挂了、你的 API Key 过期了、网络超时

现在如果这两种情况发生，Hono 会把 Node.js 的原始错误直接抛给客户端—— 或者更糟，整个进程崩溃。 你需要主动捕获这些情况，给客户端一个结构清晰的错误响应。

HTTP 状态码：错误的"语言"

REST API 用 HTTP 状态码传达"出了什么问题"。你只需要记住三个：

状态码 含义 什么时候用
200 成功 正常返回
400 客户端的锅（Bad Request） 参数缺失、格式错误——是调用方写错了
503 服务不可用（Service Unavailable） 下游 AI 调用失败——是服务端出问题了
关键区分：400 是"你的问题"，503 是"我的问题"。 前端可以根据这个决定：是提示用户"请填写必要信息"，还是"服务暂时不可用，请稍后重试"。

动手：两处改动

1
输入验证——在路由最开头检查参数

在 const { message, sessionId } = await c.req.json() 之后、sessionMap 之前插入： 

if (!message || !sessionId) {
  return c.json({ error: '缺少必填参数：message 和 sessionId' }, 400)
}

这样在参数不对时立即返回 400，不进入后续逻辑。

2
try/catch——包裹 AI 调用

把 AI 调用和后续逻辑包进 try/catch：

src/index.ts — 改动部分
// 把用户消息加入历史
history.push({ role: 'user', content: message })

try {
  const response = await client.chat.completions.create({
    model: 'deepseek-chat',
    messages: history,
  })
  const reply = response.choices[0].message.content ?? ''
  history.push({ role: 'assistant', content: reply })
  return c.json({ reply, timestamp: new Date().toISOString() })
} catch (err) {
  history.pop()  // 撤销刚才的 user 消息，避免历史被污染
  console.error('AI 调用失败', err)
  return c.json({ error: 'AI 服务暂时不可用，请稍后重试' }, 503)
}
注意 history.pop()： 如果 AI 调用失败，你刚才已经把 user 消息 push 进历史了。 如果不撤销，下次请求这个 session 时，历史里会多出一条"孤儿消息"——有用户消息但没有对应的 AI 回复，这会让对话乱掉。

完整路由对比

src/index.ts — /api/chat 完整版
app.post('/api/chat', async (c) => {
  const { message, sessionId } = await c.req.json()

  if (!message || !sessionId) {
    return c.json({ error: '缺少必填参数：message 和 sessionId' }, 400)
  }

  if (!sessionMap.has(sessionId))
    sessionMap.set(sessionId, [{ role: 'system', content: SYSTEM_PROMPT }])
  const history = sessionMap.get(sessionId)!

  history.push({ role: 'user', content: message })

  try {
    const response = await client.chat.completions.create({
      model: 'deepseek-chat',
      messages: history,
    })
    const reply = response.choices[0].message.content ?? ''
    history.push({ role: 'assistant', content: reply })
    return c.json({ reply, timestamp: new Date().toISOString() })
  } catch (err) {
    history.pop()
    console.error('AI 调用失败', err)
    return c.json({ error: 'AI 服务暂时不可用，请稍后重试' }, 503)
  }
})

验证：怎么测出错的情况

在 test.http 里加这两条——不用启动额外的东西，直接发请求：

test.http — 新增测试
### 测试 1：缺少 sessionId → 应返回 400
POST http://localhost:3000/api/chat
Content-Type: application/json

{
  "message": "你好"
}

### 测试 2：缺少 message → 应返回 400
POST http://localhost:3000/api/chat
Content-Type: application/json

{
  "sessionId": "test-001"
}
验证方法：发这两条请求，应该收到 HTTP 400 和 { "error": "缺少必填参数：message 和 sessionId" }， 而不是 500 或者空响应。

测验：巩固理解

快速测验 · 三题

1. AI 调用失败后，为什么要执行 history.pop()？

2. 用户传了一个空字符串 message: ""，你的验证 if (!message) 能拦住吗？

3. 这个接口现在服务器重启后，所有会话历史会消失。这是 bug 还是 feature？

你的对话历史现在存在内存里，重启就丢。下节课你会接入 SQLite， 把会话历史写进文件，让对话跨重启存活。 这是从"玩具"变成"真实应用"的关键一步。
