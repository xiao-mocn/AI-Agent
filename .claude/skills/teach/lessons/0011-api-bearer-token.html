第十一课：保护你的 API

Bearer Token 鉴权 · 防止 API 额度被滥用

你现在的处境

你的 AI 聊天应用已经部署在公网。这意味着： 任何知道 URL 的人都可以无限次调用它，每次调用都消耗你的 DeepSeek API 额度。

真实风险：一旦 URL 被人分享到论坛，你的 API 配额可能在几分钟内耗尽， 还可能产生账单。这不是假设——这种事每天都在发生。

本节课用最少的代码给你的 API 加上一把锁——Bearer Token 鉴权。

什么是 Bearer Token？

HTTP 协议定义了一个标准请求头 Authorization，格式是： 

Authorization: Bearer <token>

Bearer 就是"持有者"的意思——谁持有这个 token，谁就有权访问。 后端收到请求时，先检查这个 Header，如果 token 不匹配就返回 401 Unauthorized， 拒绝处理。

实践

第一步：后端加中间件 backend

在 src/index.ts 里，在所有路由之前加一个认证中间件：

// 从环境变量读取允许的 token
const ALLOWED_TOKEN = process.env.API_TOKEN

app.use('*', async (c, next) => {
  // 没有配置 token 时跳过（本地开发方便）
  if (!ALLOWED_TOKEN) return next()

  const auth = c.req.header('Authorization')
  if (auth !== `Bearer ${ALLOWED_TOKEN}`) {
    return c.json({ error: 'Unauthorized' }, 401)
  }
  return next()
})
为什么写在路由前？Hono 的中间件按注册顺序依次执行。 只要在 app.post('/chat', ...) 之前注册，每个请求都会先经过这里。

第二步：前端带上 Token frontend

在 App.vue 的 send() 函数里，给 fetch 加一个 Header：

// 从 localStorage 读 token（用户第一次访问时提示输入）
let token = localStorage.getItem('api_token')
if (!token) {
  token = prompt('请输入访问密码：') ?? ''
  if (token) localStorage.setItem('api_token', token)
}

const res = await fetch(`${API_BASE}/chat`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`,  // 新增
  },
  body: JSON.stringify({ message: text, sessionId }),
})

第三步：在 Render 加环境变量 backend

  1. 登录 Render Dashboard → 你的服务 → Environment
  2. Add Environment Variable： API_TOKEN = 一串你自己编的随机字符串（比如 my-secret-key-2024）
  3. 点击 Save → 服务自动重启
本地开发不受影响：本地没有配置 API_TOKEN 环境变量， 中间件里的 if (!ALLOWED_TOKEN) return next() 会直接放行，不需要改任何配置。

第四步：用 curl 验证

# 没有 token → 应返回 401
curl -X POST https://your-app.onrender.com/chat \
  -H "Content-Type: application/json" \
  -d '{"message":"hi","sessionId":"test"}'

# 带正确 token → 应正常返回
curl -X POST https://your-app.onrender.com/chat \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer my-secret-key-2024" \
  -d '{"message":"hi","sessionId":"test"}'

小测验

点击选项查看解析

1. 后端收到一个没有 Authorization Header 的请求，应该返回什么状态码？

  • 400 Bad Request
  • 401 Unauthorized
  • 403 Forbidden
  • 404 Not Found

401 Unauthorized：表示"没有提供有效的身份凭证"。 400 是请求格式错误，403 是有凭证但无权限，401 才是"未经认证"。

2. 我们把 token 存在 localStorage 而不是写死在代码里，主要原因是？

  • 写死在代码里会被 git 提交，任何看到仓库的人都能知道 token
  • localStorage 比变量访问更快
  • Vue 3 不支持在组件里定义常量
  • 前端变量会在每次刷新后丢失

安全考虑：硬编码的密钥会进入 git 历史，即使之后删除也能被翻出来。 敏感信息要么从环境变量读（后端），要么让用户运行时输入（前端）。

3. 为什么中间件要在路由注册之前调用 app.use()？

  • 这是 Hono 的 bug，其实顺序不影响结果
  • 只有放在最前面才能读到 Authorization Header
  • Hono 按注册顺序执行中间件，放在路由前才能拦截所有请求
  • 后注册的中间件优先级更高

Hono（和大多数 HTTP 框架一样）按照注册顺序依次执行中间件链。 先注册的先跑，所以认证中间件要最先注册，才能在路由处理之前拦截。

进阶思考

这种方案适合单人使用的工具。如果要支持多个不同用户各自的账号和对话历史， 就需要完整的用户认证系统（注册、登录、JWT）。这会是第十二课的内容。

本节要点

  • Authorization: Bearer <token> 是 HTTP 标准的认证头
  • 后端中间件在路由之前统一校验，一处代码保护所有接口
  • token 存环境变量（后端）/ localStorage（前端），不写死在代码里
  • 本地开发时不配置环境变量 → 自动放行，不影响调试
