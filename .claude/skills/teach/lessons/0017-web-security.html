一、XSS——跨站脚本攻击

攻击原理

XSS 的核心思路：攻击者把一段 恶意 JavaScript 注入到网页里，让其他用户的浏览器执行它。

假设场景：用户在聊天框输入了这段内容，然后你的前端用 innerHTML 显示它： 
<img src=x onerror="document.cookie // 窃取 Cookie，发给攻击者服务器">
浏览器渲染这段 HTML 时，图片加载失败，onerror 被执行——攻击者拿到了受害者的 Cookie。

三种 XSS 类型

类型来源典型场景
存储型（Stored）数据库恶意内容存入数据库，其他用户读取时被执行——最危险
反射型（Reflected）URL 参数恶意链接诱导用户点击，参数被直接渲染到页面
DOM 型前端 JS前端读取 URL hash 等不可信来源，直接写入 DOM

Vue 如何防 XSS

好消息：Vue 默认是安全的。双花括号插值会自动 HTML 转义：

<!-- 安全：Vue 会把 <script> 转义成 &lt;script&gt;，不会执行 -->
<p>{{ message }}</p>
危险写法：v-html 
<!-- 危险：v-html 直接插入 HTML，XSS 风险！ -->
<div v-html="aiReply"></div>
如果 aiReply 来自 AI 模型或用户输入，攻击者可以让 AI 回复一段含 <script> 的内容。

需要渲染 Markdown 怎么办？

很多 AI 聊天应用需要渲染 Markdown（代码块、加粗等）。正确做法是先解析 Markdown，再用 DOMPurify 净化 HTML：

1安装依赖 
cd ai-frontend
npm install marked dompurify
npm install -D @types/dompurify
2在组件中安全渲染 
<!-- MessageBubble.vue -->
<template>
  <div v-html="safeHtml"></div>
</template>

<script setup lang="ts">
import { computed } from 'vue'
import { marked } from 'marked'
import DOMPurify from 'dompurify'

const props = defineProps<{ content: string }>()

// marked 解析 Markdown → HTML，DOMPurify 过滤危险标签
const safeHtml = computed(() =>
  DOMPurify.sanitize(marked.parse(props.content) as string)
)
</script>
DOMPurify 会移除 <script>、onerror、javascript: 等危险内容，只保留安全的 HTML 标签。

二、CSRF——跨站请求伪造

攻击原理

经典场景（基于 Cookie）：
  1. 用户已登录 bank.com，浏览器有 session=abc Cookie
  2. 用户访问了恶意网站 evil.com
  3. evil.com 的页面里有：<img src="https://bank.com/transfer?to=attacker&amount=10000">
  4. 浏览器自动带上 Cookie 发出请求——转账成功，受害者毫不知情

为什么 JWT Bearer Token 天然免疫 CSRF？

关键区别在于：浏览器不会自动携带 Authorization 头。
  • Cookie：浏览器跨站请求时自动附带，这是 CSRF 的根源
  • Bearer Token：存在 localStorage 或 sessionStorage，只有 JS 代码主动读取后放进 Authorization: Bearer xxx 头才会发送
evil.com 的页面无法读取你 localStorage 里的 Token（受同源策略保护），所以伪造请求不带 Token，后端验证失败，攻击无效。

我们的应用已经用了 JWT Bearer Token，CSRF 这关已经过了。

如果你改用 Cookie 存 Token（比如 HttpOnly Cookie），就需要额外加 CSRF 防护：同步令牌（CSRF Token）或 SameSite Cookie 属性。

三、给 Hono 添加安全响应头

即使 XSS 和 CSRF 已有基础防护，还应该加上 HTTP 安全头，作为额外防线。

1安装 Hono 安全中间件

Hono 内置了 secureHeaders 中间件，一行搞定常用安全头：

// 已内置，不需要额外安装
import { secureHeaders } from 'hono/secure-headers'
2在 index.ts 中启用 
import { Hono } from 'hono'
import { secureHeaders } from 'hono/secure-headers'
import { cors } from 'hono/cors'

const app = new Hono()

// 安全头（放在 cors 之前）
app.use('*', secureHeaders())

// CORS 配置（限制允许的来源）
app.use('/api/*', cors({
  origin: process.env.FRONTEND_URL || 'http://localhost:5173',
  allowMethods: ['GET', 'POST', 'OPTIONS'],
  allowHeaders: ['Content-Type', 'Authorization'],
}))
3secureHeaders 默认添加的响应头
响应头作用
X-Content-Type-Options: nosniff禁止浏览器猜测 MIME 类型，防止脚本伪装成图片
X-Frame-Options: SAMEORIGIN禁止页面被嵌入 iframe，防止点击劫持
Referrer-Policy: no-referrer跨站请求不泄露来源 URL
Strict-Transport-Security强制 HTTPS（生产环境生效）
X-DNS-Prefetch-Control: off关闭 DNS 预解析，防止信息泄露

四、CORS 与安全

CORS 不是安全防护，而是浏览器的访问控制机制。理解两个关键点：

  • CORS 只约束浏览器：curl、Postman、服务器端请求完全不受 CORS 影响
  • 允许的 origin 要精确：生产环境不要用 origin: '*'，否则任何网站都能调用你的 API
// 不安全：允许所有来源
cors({ origin: '*' })

// 安全：只允许自己的前端域名
cors({ origin: 'https://your-app.onrender.com' })
生产和本地开发的 origin 不同，可以通过环境变量控制：
Render 后端添加环境变量 FRONTEND_URL=https://your-frontend.onrender.com

五、Content Security Policy（CSP）

CSP 是最强的 XSS 防线——告诉浏览器哪些来源的脚本、样式可以加载，直接从根上阻止注入脚本执行。

1通过 secureHeaders 自定义 CSP 
app.use('*', secureHeaders({
  contentSecurityPolicy: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"],
    styleSrc: ["'self'", "'unsafe-inline'"],  // 内联样式必须允许
    imgSrc: ["'self'", 'data:'],
    connectSrc: ["'self'", 'https://api.deepseek.com'],
  }
}))
CSP 可能会阻断正常功能（比如 Vue 开发模式的 eval）。建议先在 Report-Only 模式下测试，确认无误再开启强制模式。

六、本课实践：给项目加上安全头

1修改 ai-backend/src/index.ts

在现有 cors 中间件之前加上 secureHeaders：

import { secureHeaders } from 'hono/secure-headers'

// 在所有路由之前注册
app.use('*', secureHeaders())

// CORS 限制为精确的前端域名
const ALLOWED_ORIGIN = process.env.FRONTEND_URL || 'http://localhost:5173'
app.use('*', cors({
  origin: ALLOWED_ORIGIN,
  allowHeaders: ['Content-Type', 'Authorization'],
  allowMethods: ['GET', 'POST', 'OPTIONS'],
}))
2在 Render 后端添加环境变量
变量名值
FRONTEND_URL你的前端 Render 地址，如 https://ai-frontend.onrender.com
3验证安全头已生效

部署后，用浏览器 DevTools → Network → 点击任意 API 请求 → 查看 Response Headers，应该看到：

x-content-type-options: nosniff
x-frame-options: SAMEORIGIN
referrer-policy: no-referrer

七、安全检查清单

  1. 前端 不用 v-html 渲染用户输入或 AI 回复（或配合 DOMPurify 使用）✓
  2. 认证使用 JWT Bearer Token，不用 Cookie，天然免疫 CSRF ✓
  3. 后端已配置 secureHeaders 中间件 ✓
  4. CORS origin 精确指定，不使用 '*' ✓
  5. 生产 API 走 HTTPS（Render 自动配置）✓
  6. 密码使用 bcrypt 哈希存储，不存明文 ✓

八、本课总结

  • XSS 通过注入脚本攻击用户浏览器；Vue 默认安全，避免 v-html，渲染 Markdown 用 DOMPurify
  • CSRF 利用浏览器自动携带 Cookie 发起伪造请求；JWT Bearer Token 不被浏览器自动携带，天然免疫
  • secureHeaders 中间件一行代码加上多个安全响应头
  • CORS origin 精确配置，防止其他网站随意调用你的 API
  • 安全是纵深防御，不是一道墙，而是多层叠加
