第十二课：JWT 用户认证

注册与登录 · 让应用支持多用户

上节课的局限

Bearer Token 解决了"谁能用"的问题，但所有人用的是同一个 token、同一份对话历史。 真正的产品需要每个用户有自己的账号和独立数据。

这节课实现：注册 → 登录 → 拿到 JWT → 用 JWT 访问受保护接口。

JWT 是什么？

JWT（JSON Web Token）是一种自包含的令牌——用户登录成功后， 服务器把用户 ID 等信息加密签名打包成一个字符串发给前端。 之后前端每次请求都带上这个字符串，服务器验证签名后直接从中取出用户信息，不需要查数据库。

前端 POST /register
→
后端写入 users 表
→
返回成功

→
校验密码
→
返回 JWT
前端带 JWT 请求 /api/chat
→
验证 JWT 签名
→
取出 userId 处理请求

JWT 的结构

一个 JWT 长这样（三段用 . 分隔）：

eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjF9.abc123签名
段内容可以被解码？
Header算法类型（HS256）是（Base64）
Payload用户数据（userId、过期时间）是（Base64）
Signature用密钥对前两段做的签名无法伪造
JWT Payload 不是加密的，任何人都能 Base64 解码看到里面的内容。 不要把密码、信用卡等敏感信息放进去，只放 userId 等标识符。

实践

第一步：安装依赖 backend

# 在 ai-backend 目录下
npm install jsonwebtoken bcryptjs
npm install --save-dev @types/jsonwebtoken @types/bcryptjs

bcryptjs 用来对密码做哈希（绝对不能明文存密码）；jsonwebtoken 用来签发和校验 JWT。

第二步：数据库加 users 表 SQLite

在 src/db.ts 里加：

// 创建 users 表
db.prepare(`
  CREATE TABLE IF NOT EXISTS users (
    id       INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT    UNIQUE NOT NULL,
    password TEXT    NOT NULL
  )
`).run()

export const createUser = db.prepare(
  'INSERT INTO users (username, password) VALUES (?, ?)'
)
export const findUser = db.prepare(
  'SELECT * FROM users WHERE username = ?'
)

第三步：注册和登录接口 backend

import jwt from 'jsonwebtoken'
import bcrypt from 'bcryptjs'
import { createUser, findUser } from './db'

const JWT_SECRET = process.env.JWT_SECRET ?? 'dev-secret'

// 注册
app.post('/register', async (c) => {
  const { username, password } = await c.req.json()
  if (!username || !password) return c.json({ error: '缺少参数' }, 400)

  const hash = await bcrypt.hash(password, 10)
  try {
    createUser.run(username, hash)
    return c.json({ ok: true })
  } catch {
    return c.json({ error: '用户名已存在' }, 409)
  }
})

// 登录
app.post('/login', async (c) => {
  const { username, password } = await c.req.json()
  const user = findUser.get(username) as any
  if (!user) return c.json({ error: '用户不存在' }, 401)

  const ok = await bcrypt.compare(password, user.password)
  if (!ok) return c.json({ error: '密码错误' }, 401)

  const token = jwt.sign({ userId: user.id }, JWT_SECRET, { expiresIn: '7d' })
  return c.json({ token })
})

第四步：把 Bearer Token 中间件换成 JWT 校验 backend

app.use('/api/*', async (c, next) => {
  const auth = c.req.header('Authorization') ?? ''
  const token = auth.replace('Bearer ', '')
  try {
    const payload = jwt.verify(token, JWT_SECRET) as { userId: number }
    c.set('userId', payload.userId)  // 传给后续路由
    return next()
  } catch {
    return c.json({ error: '无效 token' }, 401)
  }
})
注意路径改为 /api/*：注册和登录接口本身不需要鉴权，只保护 /api/ 下的接口。

第五步：在 Render 加 JWT_SECRET 环境变量 backend

JWT_SECRET 是签名密钥，必须保密——任何知道这个值的人都能伪造任意用户的 token。 设置一个长度 32 位以上的随机字符串，不要用 dev-secret。

用 curl 测试

# 注册
curl -X POST http://localhost:3000/register \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"123456"}'

# 登录，拿到 token
curl -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"123456"}'

# 用 token 发消息
curl -X POST http://localhost:3000/api/chat \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <上面拿到的token>" \
  -d '{"message":"你好","sessionId":"s1"}'

小测验

点击选项查看解析

1. 为什么密码要用 bcrypt 哈希，而不是直接存明文？

  • bcrypt 比明文存储更省空间
  • 数据库一旦泄露，攻击者也无法还原出原始密码
  • bcrypt 加密后密码可以被解密，方便找回
  • 这是 Node.js 的强制要求

单向哈希：bcrypt 是不可逆的哈希，无法从哈希值还原密码。 即使数据库被拖库，攻击者也只能拿到哈希值，无法直接登录其他网站（防止撞库攻击）。

2. JWT Payload 里的数据安全吗？

  • 完全安全，JWT 是加密的
  • 只有服务端能读取
  • 任何人都能 Base64 解码读到，但无法伪造签名
  • 浏览器会自动加密存储

JWT Payload 只是 Base64 编码（不是加密），任何人都能解码。 安全性来自签名——没有密钥就无法伪造有效 token。 所以 Payload 里只放 userId，不放密码等敏感信息。

3. 为什么认证中间件挂在 /api/* 而不是 *？

  • 注册/登录接口本身不需要 token，挂 * 会导致无法登录
  • /api/* 性能更好
  • Hono 不支持 * 通配符
  • 这只是代码风格约定，效果相同

如果把鉴权挂在所有路径（*），用户连 /login 都需要 token 才能访问—— 但 token 是登录后才能拿到的，形成死锁。所以只保护需要认证的路径。

本节要点

  • JWT = 自包含令牌，服务器无需查库即可验证身份
  • 密码必须用 bcrypt 哈希存储，绝不明文
  • JWT_SECRET 放环境变量，生产环境必须是强随机字符串
  • 认证中间件只保护需要登录的路径（/api/*），不拦截登录/注册接口
